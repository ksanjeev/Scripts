Descripción general

Puede instalar el software Managed System por medio de uno de los dos métodos. El CD Dell™ PowerEdge™ Installation and Server Management incluye secuencias de comandos de instalación y paquetes RPM para instalar, actualizar y desinstalar Dell OpenManage™ Server Administrator y otros componentes del software Managed System en el sistema administrado. Además, Server Administrator se puede instalar en múltiples sistemas en una red mediante una instalación desatendida.

El primer método de instalación utiliza la secuencia de comandos de instalación personalizada que se incluye, srvadmin-install.sh. Esta secuencia de comandos permite la instalación rápida desatendida y la instalación personalizada, desatendida o interactiva. Al incluir la secuencia de comandos srvadmin-install.sh en las secuencias de comandos de Linux se puede instalar Server Administrator en uno o varios sistemas, en modo interactivo o desatendido, así como de manera local o a través de la red.

El segundo método de instalación utiliza los paquetes RPM de Server Administrator que se incluyen en los directorios personalizados y el comando rpm de Linux. Esto permite la instalación interactiva personalizada. Se pueden escribir secuencias de comandos de Linux que instalen Server Administrator en uno o varios sistemas por medio de una instalación desatendida de manera local o a través de la red.

No se recomienda combinar los dos métodos de instalación y tal combinación puede requerir que usted instale manualmente los paquetes RPM necesarios de Server Administrator que se incluyen en los directorios personalizados, usando el comando rpm de Linux.

NOTA: Consulte el archivo readme_ins.txt en el CD Dell PowerEdge Installation and Server Management para ver una lista de los sistemas admitidos actualmente.

Instalación desatendida y silenciosa con secuencia de comandos

Puede usar el CD Dell PowerEdge Installation and Server Management para realizar una instalación desatendida y silenciosa del software Managed System con una secuencia de comandos por medio de la línea de comandos (usando paquetes RPM) en los sistemas que ejecutan sistemas operativos Red Hat Enterprise Linux y SUSE Linux Enterprise Server admitidos.

  • Lea el archivo de instalación de Dell OpenManage readme_ins.txt en el CD Dell PowerEdge Installation and Server Management. Este archivo contiene la información más reciente acerca de las nuevas funciones, correcciones y requisitos de hardware, además de la información de los problemas conocidos.

  • Lea las instrucciones para la instalación del sistema operativo.

Requisitos para la instalación

En las secciones siguientes se describen los requisitos generales de Managed System Software.

Versiones de los sistemas operativos admitidos

Managed System Software se ejecuta como mínimo en los siguientes sistemas operativos:

  • Red Hat Enterprise Linux AS, ES y WS (versión 3) x86, actualización 6

  • Red Hat Enterprise Linux AS, ES y WS, (versión 3) para la tecnología Intel® de memoria extendida 64 (Intel EM64T), actualización 6

  • Red Hat Enterprise Linux AS, ES y WS (versión 4) para Intel x86

  • Red Hat Enterprise Linux AS, ES y WS (versión 4) para Intel EM64T

  • SUSE Linux Enterprise Server (versión 9) SP3 para Intel EM64T

NOTA: Consulte el archivo "léame" de Server Administrator en el CD Dell PowerEdge Installation and Server Management o la Guía de compatibilidad de Server Administrator de Dell OpenManage en el CD Documentation para obtener la lista detallada más reciente de los servicios de Server Administrator que son compatibles con cada sistema operativo admitido.

Requisitos del sistema

Managed System Software debe estar instalado en cada uno de los sistemas que se van a administrar. Se pueden administrar entonces todos los sistemas que ejecutan el Managed System Software en forma local o remota mediante un explorador web admitido.

Requisitos del sistema administrado

  • Una de las versiones admitidas de los sistemas operativos.

  • Mínimo de 64 MB de RAM.

  • Mínimo de 256 MB de espacio libre en la unidad de disco duro.

  • Derechos de administrador.

  • Conexión TCP/IP en el sistema supervisado y en el sistema remoto para facilitar la administración de sistemas remotos.

  • El protocolo simple de gestión de red (SNMP).

  • Mouse, teclado y monitor para administrar un sistema localmente. El monitor requiere una resolución de pantalla mínima de 800 x 600. La configuración para la resolución recomendada es 1024 x 768.

  • El servicio de acceso remoto de Server Administrator requiere que haya un Remote Access Controller (RAC) instalado en el sistema que se va a administrar.

NOTA: Consulte la Guía del usuario de Dell Remote Access Controller 4, la Guía del usuario de Dell Remote Access Controller 5 o la Guía del usuario del controlador de acceso remoto incorporado/MC de Dell para ver los requisitos completos de software y de hardware.
  • El Storage Management Service de Server Administrator requiere que Dell OpenManage Storage Management está instalado en el sistema para que pueda ser administrado correctamente. Consulte la Guía del usuario de Dell OpenManage Server Administrator para ver la lista completa de los requisitos de software y hardware.

Estándares de protocolos de Systems Management admitidos

Un protocolo estándar admitido debe estar instalado en el sistema administrado antes de instalar el Server Administrator. En los sistemas operativos Red Hat Enterprise Linux y SUSE Linux Enterprise Server compatibles, Server Administrator sólo admite el estándar SNMP de administración de sistemas. Se debe instalar el paquete SNMP proporcionado con el sistema operativo. El CIM y el WMI no están disponibles.

NOTA: Para obtener más información sobre la instalación de un protocolo de Systems Management admitido en el sistema administrado, consulte la documentación del sistema operativo.

Compatibilidad con el núcleo dinámico (DKS)

Server Administrator proporciona controladores de dispositivos precompilados para los núcleos precompilados enumerados en el archivo "léame" de Server Administrator del CD Dell PowerEdge Installation and Server Management. Si el núcleo en ejecución no es uno de los núcleos precompilados listados en el archivo léame, o si el núcleo en ejecución se reconfigura y se recompila de forma que ninguno de los controladores de dispositivos precompilados de Server Administrator admita el núcleo recompilado, es posible que Server Administrator deba utilizar la función de compatibilidad con el núcleo dinámico para admitir el núcleo en ejecución.

Si aparece el siguiente mensaje durante el inicio de los controladores de dispositivos de Server Administrator, significa que Server Administrator intentó utilizar la función de compatibilidad con el núcleo dinámico, pero no pudo hacerlo porque no se cumplieron ciertos requisitos previos:

Building <driver> using DKS... [FAILED]

(Generando <archivo controlador> con DKS... [FALLÓ])

donde <archivo controlador> es dcdbas o dell_rbu

NOTA: Server Administrator registra los mensajes en el archivo de registro /var/log/messages.

Para usar la compatibilidad con el núcleo dinámico (DKS), se debe identificar el núcleo que se está ejecutando y después verificar los requisitos previos del mismo.

Determinación del núcleo en ejecución

  1. Iniciar una sesión como root.

  2. Escriba el comando siguiente en la consola y presione <Entrar>:

uname -r

El sistema muestra un mensaje que identifica el núcleo en ejecución. Si no es uno de los que aparecen en la lista del archivo "léame" de Managed System Software, es posible que Managed System Software deba usar la DKS para admitirlo.

Requisitos previos de la compatibilidad con el núcleo dinámico

Para que Managed System Software pueda usar la DKS, las siguientes dependencias se deben cumplir antes de iniciar Server Administrator.

  • El núcleo en ejecución debe admitir módulo cargable activado.

  • El origen para construir módulos de núcleo para el núcleo en ejecución debe estar disponible a partir de /lib/modules/`uname -r`/build. En sistemas que ejecutan Red Hat Enterprise Linux (versión 3) y SUSE Linux Enterprise Server (versión 9), el RPM kernel-source proporciona la fuente necesaria del núcleo. En sistemas que ejecutan Red Hat Enterprise Linux (versión 4), los RPM kernel*-devel proporcionan el origen necesario del núcleo para generar módulos de núcleo.

  • La utilidad para la creación de GNU debe estar instalada. El RPM make proporciona esta utilidad.

  • El compilador GNU C (gcc) debe estar instalado. El RPM gcc proporciona este compilador.

  • El vinculador GNU (ld) debe estar instalado. El RPM binutils proporciona este vinculador.

Cuando se hayan cumplido estos requisitos previos, la compatibilidad con el núcleo dinámico (DKS) generará automáticamente un controlador de dispositivos cuando sea necesario durante el inicio de Server Administrator.

NOTA: Los núcleos no admitidos son aquellos núcleos que no son compatibles con un controlador de dispositivos precompilado. Si usted está ejecutando un núcleo admitido, vea "Instalación y actualización de Managed System Software".

Uso de la compatibilidad con el núcleo dinámico después de la instalación de Server Administrator

Para permitir que Server Administrator admita un núcleo no admitido por un controlador de dispositivos precompilado y que se ha cargado después de instalar el Server Administrator, realice los pasos siguientes:

  1. Asegúrese de que los requisitos previos de DKS se cumplan en el sistema que se va a administrar.

  2. Inicie el nuevo núcleo en el sistema.

Server Administrator crea un controlador de dispositivos para el núcleo que se está ejecutando en el sistema la primera vez que se inicia después de cargar el núcleo. De manera predeterminada, el Server Administrator se inicia al iniciar el sistema.

Copia de un controlador de dispositivos creado dinámicamente en sistemas que ejecutan el mismo núcleo

Cuando Server Administrator genera dinámicamente un controlador de dispositivo para el núcleo en ejecución, lo instala en el directorio /lib/modules/<núcleo>/kernel/drivers/firmware, donde <núcleo> es el nombre del núcleo (que se obtiene al escribir uname -r). Si tiene un sistema que ejecuta el mismo núcleo para el que se generó un controlador de dispositivo, puede copiar el controlador de dispositivo recién generado al directorio /var/omsa/dks/<núcleo> en el otro sistema para que lo use Server Administrator. Esta acción permite que Server Administrator utilice la función de DKS en varios sistemas sin tener que instalar el origen del núcleo en cada uno de ellos.

Un ejemplo es el escenario siguiente: el sistema A ejecuta un núcleo que no es compatible con uno de los controladores de dispositivos precompilados de Server Administrator. El sistema B ejecuta el mismo núcleo. Realice los siguientes pasos para crear un controlador de dispositivo en el sistema A y copiarlo en el sistema B para que lo utilice Server Administrator:

  1. Asegúrese de que los requisitos previos de DKS se cumplan en el sistema A.

  2. Inicie el Server Administrator en el sistema A.

Server Administrator crea un controlador de dispositivos para el núcleo que se está ejecutando en el sistema A durante el inicio.

  1. Escriba uname -r en el sistema A para determinar el nombre del núcleo en ejecución.

  2. Copie todos los archivos dcdbas.* o dell_rbu.* en el directorio /lib/modules/<núcleo>/kernel/drivers/firmware del sistema A al directorio /var/omsa/dks/<núcleo> del sistema B, donde <núcleo> es el nombre de núcleo que se obtuvo escribiendo uname -r en el paso 3.

NOTA: Es posible que el directorio /lib/modules/<núcleo>/kernel/drivers/firmware contenga uno o varios de los archivos siguientes: dcdbas.* o dell_rbu.*
NOTA: Es posible que deba crear el directorio /var/omsa/dks/<núcleo> en el sistema B. Por ejemplo, si el nombre del núcleo es 1.2.3-4smp, usted puede crear el directorio escribiendo mkdir -p /var/omsa/dks/1.2.3-4smp.
  1. Inicie el Server Administrator en el sistema B.

Server Administrator detecta que el controlador de dispositivo que copió en el directorio /var/omsa/dks/<núcleo> admite el núcleo en ejecución y utiliza ese controlador de dispositivo.

NOTA: También se puede usar este procedimiento cuando se actualiza Server Administrator si la nueva versión del mismo no admite el núcleo en ejecución con un controlador de dispositivos precompilado.
NOTA: Cuando haya desinstalado Server Administrator del sistema B, no se eliminarán los archivos /var/omsa/dks/<núcleo>/*. que usted copió al sistema B. Usted debe eliminar los archivos si ya no se requieren.

Cómo forzar la compatibilidad con el núcleo dinámico en las versiones de actualización de Red Hat Enterprise Linux cuando el núcleo está dañado

Server Administrator proporciona controladores de dispositivos precompilados para las versiones principales de los sistemas operativos compatibles Red Hat Enterprise Linux. Red Hat Enterprise Linux es compatible con la carga de controladores de dispositivos creados para la versión principal, en las versiones de actualización. Esto significa que Server Administrator no tiene que enviar controladores de dispositivos precompilados para cada versión de actualización de Red Hat Enterprise Linux y los usuarios no están obligados a usar el DKS para ejecutar Server Administrator en cada sistema que ejecuta una versión de actualización de Red Hat Enterprise Linux. Sin embargo, si se carga de un controlador de dispositivos que fue creado para la versión principal de Red Hat Enterprise Linux (versión 3) en una versión de actualización, es posible que se dañe el núcleo. Si el núcleo de un sistema que ejecuta una versión de actualización de Red Hat Enterprise Linux (versión 3) ha resultado dañado por este proceso de carga de controlador de dispositivo, se puede usar el comando restart-forcekernelmatch de la secuencia de comandos de inicio de Server Administrator para forzar el uso del DKS en esta situación. El DKS generará controladores de dispositivos que no dañen el núcleo en ejecución.

Cómo determinar si el núcleo en ejecución está dañado

Después de que los servicios de Server Administrator se hayan iniciado, realice los siguientes pasos en las versiones de actualización de Red Hat Enterprise Linux para determinar si el núcleo ha sido dañado:

  1. Iniciar una sesión como root.

  2. Ejecute el comando siguiente:

lsmod

Si usted ve Dañado: GF en la primera línea del mensaje, como se muestra en el mensaje siguiente, el núcleo en ejecución está dañado:

Module Size Used by Tainted: GF

(Tamaño del módulo utilizado por Dañado: GF)

El estado "dañado" puede deberse al proceso de carga del controlador de dispositivos de Server Administrator.

Cómo forzar la compatibilidad con el núcleo dinámico en las versiones de actualización de Red Hat Enterprise Linux

Después de la instalación de Server Administrator, realice los pasos a continuación para forzar el uso del DKS en las versiones de actualización de Red Hat Enterprise Linux a fin de generar controladores de dispositivos para el núcleo en ejecución, si son necesarios, de manera que estos no dañen el núcleo:

  1. Asegúrese que se cumplan los requisitos previos del DKS.

  2. Ejecute el comando siguiente:

/etc/init.d/instsvcdrv restart-forcekernelmatch

Este comando detendrá primero los controladores de dispositivos de Server Administrator. Después realizará una búsqueda de los controladores de dispositivos precompilados a cargar, revisando los controladores de dispositivos precompilados que fueron generados para un núcleo cuyo nombre sea una coincidencia exacta del nombre del núcleo en ejecución. Si no encuentra una coincidencia exacta, utilizará el DKS para generar controladores de dispositivos para el núcleo en ejecución. Finalmente, el comando reiniciará los controladores de dispositivos de Server Administrator.

NOTA: El sistema debe ser reiniciado para borrar el estado "dañado" del núcleo.

Controlador de dispositivos OpenIPMI

El servicio Instrumentation Service de Server Administrator necesita del controlador de dispositivos OpenIPMI para poder proporcionar la información y las funciones basadas en IPMI. También requiere de una versión mínima del controlador de dispositivos OpenIPMI. La versión mínima que se requiere está en base al sistema operativo Red Hat Enterprise Linux o SUSE Linux Enterprise Server.

Server Administrator revisa y aplica este requisito en tres instancias. Dos de las instancias son durante la instalación y la tercera es un paso del tiempo de ejecución. Las instancias son:

  • Durante la instalación personalizada de Server Administrator por medio de la secuencia de comandos de shell srvadmin-install.sh.

  • Durante la instalación de Server Administrator mediante RPM.

  • Al tiempo de ejecución, cuando se inicia Server Administrator Instrumentation Service.

Bloqueo durante la instalación personalizada de Server Administrator con la secuencia de comandos de shell srvadmin-install.sh

Durante la instalación personalizada de Server Administrator por medio de la secuencia de comandos de shell srvadmin-install.sh, esta última ejecuta automáticamente otra secuencia de comandos de shell denominada srvadmin-openipmi.sh, para asegurarse que ya esté instalada en el sistema una versión aceptable del controlador de dispositivos OpenIPMI (por ejemplo, una versión que sea igual o mayor que la versión mínima del controlador de dispositivos OpenIPMI requerida por Server Administrator Instrumentation Service). Si no se encuentra una versión aceptable, la secuencia de comandos de shell srvadmin-install.sh ejecuta nuevamente la secuencia de comandos de shell srvadmin-openipmi.sh para intentar la instalación de una versión aceptable. Si el intento falla por cualquier motivo, aparecerá un mensaje de error correspondiente con instrucciones que describen cómo resolver el error. Usted debe resolver los errores que haya antes de ejecutar nuevamente la secuencia de comandos de shell srvadmin-install.sh para instalar Server Administrator.

Bloqueo durante la instalación de Server Administrator con RPM

Para ayudar a seguir el orden correcto de la instalación con RPM, el RPM srvadmin-hapi depende de uno de los RPM srvadmin-ipmi. Cada RPM srvadmin-ipmi está destinado a una distribución específica y depende de un RPM de OpenIPMI para una distribución específica. El RPM srvadmin-hapi no se instalará a menos que se cumplan las dependencias del RPM.

Durante la instalación de Server Administrator por medio de RPM, el RPM srvadmin-hapi revisará si ya está instalada en el sistema una versión aceptable del controlador de dispositivos OpenIPMI. Si no se encuentra una versión aceptable, el RPM srvadmin-hapi no podrá instalarse y mostrará un mensaje de error indicando que usted debe instalar o actualizar el controlador de dispositivos OpenIPMI. En este caso, los RPM que dependen del RPM srvadmin-hapi tampoco podrán instalarse. En dichos casos, usted deberá instalar o actualizar el archivo controlador OpenIPMI por medio de los siguientes pasos adicionales:

  1. Navegue al directorio srvadmin/linux/supportscripts.

  2. Ejecute la siguiente secuencia de comandos de shell srvadmin-openipmi.sh:

sh srvadmin-openipmi.sh install

Intente instalar nuevamente el RPM srvadmin-hapi y los demás RPM que no se pudieron instalar debido a que dependen del RPM srvadmin-hapi. Realice los pasos a continuación:

  1. Desplácese al directorio srvadmin/linux/custom/* en el que estaba anteriormente.

  2. Ejecute el siguiente comando para ver qué paquetes de Server Administrator se instalaron bien anteriormente.

rpm -qa | grep srvadmin

La consulta muestra una lista de todos los paquetes instalados de Server Administrator.

  1. Instale cada uno de los RPM en el directorio en el que no pudieron instalarse, debido a que dependen del RPM srvadmin- hapi que tampoco pudo instalarse, ejecutando el siguiente comando:

rpm –ihv srvadmin-hapi*.rpm srvadmin-isvc*.rpm srvadmin-omhip*.rpm srvadmin-<los_demás_paquetes>*.rpm

Alternativamente, durante la instalación del RPM srvadmin-hapi con RPM, si ya está instalado en el sistema el RPM de OpenIPMI para una versión del archivo controlador OpenIPMI que cumpla con la versión mínima del archivo controlador OpenIPMI requerido por Server Administrator, pero los módulos de controlador de OpenIPMI aún no han sido generados e instalados para la versión dada del archivo controlador OpenIPMI debido a que el RPM kernel-source se necesita pero no está instalado actualmente, el RPM srvadmin-hapi no se podrá instalar y mostrará un mensaje de error. El mensaje de error le indicará que debe instalar el RPM kernel-source y generar el archivo controlador de OpenIPMI. En este caso, todos los RPM que dependen del RPM srvadmin-hapi tampoco podrán instalarse. Usted deberá instalar entonces el RPM kernel-source, y generar e instalar los módulos de controlador de OpenIPMI, realizando los siguientes pasos adicionales:

  1. Para los sistemas que ejecutan Red Hat Enterprise Linux (versión 3), consiga una copia del RPM kernel-source adecuada para ejecutar el núcleo a partir de los CD de instalación del sistema operativo (o de alguna otra fuente).

En sistemas que ejecutan Red Hat Enterprise Linux (versión 4) y SUSE Linux Enterprise Server, los RPM kernel*-devel proporcionan la fuente necesaria para generar módulos de núcleo.

  1. Instale el RPM kernel-source o los RPM kernel*-devel ejecutando el siguiente comando desde donde encontró o copió los RPM:

rpm –ihv kernel*.rpm

  1. Navegue al directorio srvadmin/linux/supportscripts.

  2. Ejecute la secuencia de comandos de shell srvadmin-openipmi.sh como se muestra a continuación. Esto generará e instalará los módulos de controlador de OpenIPMI.

sh srvadmin-openipmi.sh install

Intente instalar nuevamente el RPM srvadmin-hapi y todos los RPM que no se pudieron instalar debido a que dependen del RPM srvadmin-hapi.

Degradación de la funcionalidad cuando se inicia Server Administrator Instrumentation Service

Después de que Server Administrator está instalado, Instrumentation Service de Server Administrator realizará siempre una revisión de tiempo de ejecución del controlador de dispositivos OpenIPMI al iniciar. Server Administrator Instrumentation Service se inicia siempre que usted ejecute el comando de reinicio srvadmin-services.sh o srvadmin-services.sh, o cuando se reinicie el sistema (reinicio durante el cual se iniciará Server Administrator Instrumentation Service).

Aunque la instalación de Server Administrator se bloquea cuando no hay una versión aceptable del controlador de dispositivos OpenIPMI ya instalada en el sistema, existe aún la posibilidad, aunque no es común, que un usuario pueda desinstalar o reemplazar una versión aceptable del controlador de dispositivos OpenIPMI después de que se instaló Server Administrator. Server Administrator no puede prevenir que esto ocurra.

Para detectar si un usuario ha desinstalado o reemplazado una versión aceptable del controlador de dispositivos OpenIPMI después de que Server Administrator ha sido instalado, Instrumentation Service de Server Administrator siempre revisa la versión del controlador de dispositivos OpenIPMI al iniciar. Si no se encuentra una versión aceptable del controlador de dispositivos OpenIPMI, Instrumentation Service de Server Administrator se degradará a sí mismo para no proporcionar ninguna información o funcionalidad basadas en IPMI. Esto significa, principalmente, que no proporcionará ningún dato de sondas (por ejemplo, datos de sondas de ventiladores, de temperaturas y de voltaje).

Si sabe o sospecha que el controlador de dispositivos OpenIPMI del sistema no es aceptable, puede ejecutar la secuencia de comandos de shell srvadmin-openipmi.sh en el CD Dell PowerEdge Installation and Server Management para instalar una versión aceptable realizando nuevamente los pasos siguientes:

  1. Navegue al directorio srvadmin/linux/supportscripts.

  2. Ejecute la secuencia de comandos de shell srvadmin-openipmi.sh como se muestra a continuación.

sh srvadmin-openipmi.sh install

Si no está seguro si el controlador de dispositivos OpenIPMI del sistema es aceptable, ejecute el comando srvadmin-openipmi.sh status en vez del anterior. En vez de intentar la instalación o actualización del controlador de dispositivos OpenIPMI, que es lo que hace el comando srvadmin-openipmi.sh install, si es necesario, el comando srvadmin-openipmi.sh status le informa si el controlador de dispositivos OpenIPMI del sistema es aceptable o no. Después de ejecutar el comando de estado srvadmin-openipmi.sh, usted puede decidir entonces si necesita y desea ejecutar el comando srvadmin-openipmi.sh install.

Instalación y actualización de Managed System Software

Esta sección explica como instalar y actualizar el Managed System Software usando las siguientes opciones de instalación:

  • Utilice la secuencia de comandos de shell srvadmin-install.sh para las instalaciones rápidas o personalizadas, en modo interactivo o desatendido

  • Utilice los comandos RPM para las instalaciones personalizadas, en modo interactivo o desatendido

Requisitos previos para instalar Managed System Software

  • Se debe iniciar la sesión como root.

  • El núcleo en ejecución debe admitir módulo cargable activado.

  • El directorio /opt debe tener al menos 250 MB de espacio libre y los directorios /tmp y /var deben tener al menos 20 MB de espacio libre cada uno. Si decide usar un directorio no predeterminado para la instalación, éste también debe tener al menos 250 MB de espacio libre.

  • El paquete ucd-snmp o net-snmp que se incluye con el sistema operativo se deberá instalar si usted utiliza SNMP para administrar el servidor. Si desea utilizar agentes de asistencia para el agente ucd-snmp o net-snmp, debe instalar la asistencia del sistema operativo para el estándar SNMP antes de instalar Server Administrator. Para más información sobre la instalación de SNMP, consulte las instrucciones de instalación del sistema operativo que se esté ejecutando en el sistema.

NOTA: Al instalar un paquete RPM en Red Hat Enterprise Linux o SUSE Linux Enterprise Server, para evitar las advertencias acerca de la clave RPM-GPG, importe la clave con un comando similar al siguiente:

rpm --import /mnt/cdrom/srvadmin/linux/RPM-GPG-KEY

  • Debe instalar todos los RPM requeridos como requisito previo para una instalación satisfactoria.

Si su sistema viniera con un sistema operativo Red Hat Enterprise Linux (versiones 3 y 4) o SUSE Linux Enterprise Server (versión 9) instalados de fábrica, consulte el archivo léame de la instalación de Server Administrator (readme_ins.txt) en el CD Dell PowerEdge Installation and Server Management para obtener información sobre los RPM que se deben instalar manualmente antes de instalar el software Managed System. En la mayoría de los casos usted no necesitará instalar manualmente los RPM. Para asegurarse, consulte el archivo léame.

Si su sistema no viniera con el sistema operativo de Linux instalado de fábrica, y usted no instaló un sistema operativo Red Hat Enterprise Linux (versiones 3 y 4) o SUSE Linux Enterprise Server (versión 9) por medio de Server Assistant, entonces usted tendrá que instalar manualmente los RPM de prerrequisito antes de la instalación del software Managed System. Para su comodidad, estos archivos RPM se incluyen en el paquete o CD Dell PowerEdge Installation and Server Management. Diríjase al subdirectorio /srvadmin/linux/RPMS/sistema_operativo de la versión específica del sistema operativo Red Hat Enterprise Linux (versiones 3 y 4) o SUSE Linux Enterprise Server (versión 9) y ejecute
rpm -Uvh * para instalar estos archivos RPM antes de instalar el software Managed System.

Utilidad de instalación personalizada de Server Administrator

Esta utilidad se ejecutará en el modo interactivo si usted no especifica ninguna opción y se ejecutará silenciosamente si proporciona una o más opciones.

Instalación desatendida del software de sistema administrado

Puede usar las opciones en el CD Dell PowerEdge Installation and Server Management Instalación rápida e Instalación personalizada para el procedimiento de instalación desatendida.

La instalación desatendida permite instalar simultáneamente el Server Administrator en varios sistemas. Se puede realizar una instalación desatendida creando un paquete de instalación desatendida que contenga todos los archivos de software del sistema administrado que se necesitan.

El paquete de instalación desatendida se distribuye a los sistemas remotos mediante una herramienta de distribución de software de un proveedor independiente de software. Después de que se distribuya el paquete, el RPM instalará el software.

El paquete de instalación desatendida personalizada se encuentra en el directorio que usted creó según lo descrito en la sección "Utilización de la secuencia de comandos de Shell para realizar la instalación personalizada en modo interactivo". Este directorio contiene todos los RPM para los componentes del software Managed System que se van a distribuir.

  1. Configure el software de distribución de software, que obtuvo de un proveedor independiente de software, para ejecutar el comando rpm -1 *.rpm una vez que el paquete de instalación desatendida se ha distribuido.

  2. Utilice el software de distribución de un proveedor independiente de software para distribuir el paquete de instalación desatendida a sistemas remotos. El comando RPM instala Server Administrator en cada sistema remoto.

Revisión de dependencias

El RPM tiene una función de prueba que verifica las dependencias de software sin instalar realmente el software. Para ejecutar esta verificación dependencias, escriba rpm -ihv *.rpm –-test. Este comando es válido para todos los tipos de instalación.

NOTA: La función del comando rpm –-test no realiza ninguna verificación del hardware. Sólo verificará las dependencias de software del RPM.

La opción de instalación desatendida Instalación rápida utiliza los subdirectorios /srvadmin/linux/supportscripts y los subdirectorios /srvadmin/linux/RPMS del CD Dell PowerEdge Installation and Server Management como el paquete de instalación desatendida. Cuando se ejecuta el software del proveedor independiente (ISV), éste ejecuta los RPM para instalar Server Administrator en cada sistema remoto.

NOTA: Si no hay una versión aceptable del archivo controlador OpenIPMI ya instalada en los sistemas remotos, el RPM srvadmin-hapi no podrá instalarse y mostrará un mensaje de error informando que se debe instalar o actualizar el archivo controlador OpenIPMI. En este caso, todos los RPM que dependen del RPM srvadmin-hapi tampoco podrán instalarse. En estos casos, usted debe distribuir también los archivos RPM relacionados con OpenIPMI en el subdirectorio srvadmin/linux/custom/add-openipmi-sistema_operativo que se aplique a su sistema operativo. Actualización de la versión 4.3 y posteriores

Para las versiones 4.3 y posteriores del software de Dell OpenManage, puede actualizar su sistema usando comandos RPM o la secuencia de comandos de shell srvadmin-install.sh. Asegúrese que todos los componentes instalados estén actualizados cuando realice cualquiera de estos procedimientos.

NOTA: Se recomienda usar la secuencia de comandos de instalación, srvadmin-install.sh.
Uso de los RPM
  1. Inicie sesión como root en el sistema que ejecuta Red Hat Enterprise Linux o SUSE Linux Enterprise Server que requiere los componentes de sistema administrado actualizados.

  2. Inserte el CD Dell PowerEdge Installation and Server Management en la unidad de CD del sistema.

Si el CD no se monta automáticamente, use un comando similar al siguiente:

mount /dev/cdrom /mnt/cdrom

mount /dev/cdrom /media/cdrom

  1. Después de que el CD se monta, desplácese al directorio RPMS usando un comando similar al siguiente:

cd /mnt/cdrom/srvadmin/linux/RPMS

cd /media/cdrom/srvadmin/linux/RPMS

  1. Actualice el sistema Red Hat Enterprise Linux o SUSE Linux Enterprise Server con un solo comando grande o con archivos RPM individuales. El comando sencillo largo es una combinación de RPM y se puede ejecutar escribiendo sólo un comando.

Para actualizar todos los paquetes de instalación previos con un comando sencillo, escriba:

rpm -Uhv `rpm -q --queryformat "%{NOMBRE}* " \`rpm -qa | grep srvadmin\``

Para actualizar todos los paquetes instalados previamente usando RPM individuales, realice los pasos siguientes:

    1. Consulte la base de datos de rpm para todos los paquetes srvadmin instalados escribiendo:

rpm -qa | grep srvadmin

La consulta muestra una lista de todos los paquetes instalados, incluyendo los existentes o versiones anteriores.

Por ejemplo:

srvadmin-omilcore-4.3.0-1.386.rpm
srvadmin-hapi-4.3.0-1.386.rpm
srvadmin-deng-4.3.0-1.386.rpm
srvadmin-isvc-4.3.0-1.386.rpm
...

    1. Escriba el comando de actualización RPM, incorporando los paquetes devueltos en el paso a. Este comando actualiza los componentes de software de Dell OpenManage instalados.

Por ejemplo:

rpm -Uhv srvadmin-omilcore-4.3.0-1.386.rpm srvadmin-hapi-4.3.0-1.386.rpm srvadmin-deng-4.3.0-1.386.rpm srvadmin-isvc-4.3.0-1.386.rpm

Si el archivo controlador OpenIPMI no está instalado actualmente en el sistema, o si la versión del archivo controlador OpenIPMI que está instalada en el sistema es inferior a la versión mínima requerida por Server Administrator, el RPM srvadmin-hapi no podrá actualizarse y mostrará un mensaje de error informando que se debe instalar o actualizar el archivo controlador OpenIPMI. En este caso, todos los RPM que dependen del RPM srvadmin-hapi tampoco podrán actualizarse. En dichos casos, usted deberá instalar o actualizar el archivo controlador OpenIPMI por medio de los siguientes pasos adicionales:

  1. Navegue al directorio srvadmin/linux/supportscripts.

  2. Ejecute la secuencia de comandos de shell srvadmin-openipmi.sh como se muestra a continuación.

sh srvadmin-openipmi.sh install

Intente actualizar nuevamente el RPM srvadmin-hapi y los demás RPM que no se pudieron actualizar debido a que dependen del RPM srvadmin-hapi. Realice los pasos a continuación:

  1. Regrese al directorio en el que estaba anteriormente.

  2. Ejecute los siguientes comandos para ver qué paquetes de Server Administrator se actualizaron bien anteriormente.

rpm -qa | grep srvadmin

La consulta muestra una lista de todos los paquetes instalados de Server Administrator. Los que ya estén actualizados mostrarán el número de la nueva versión. Los que aún no estén actualizados mostrarán el número de la versión anterior.

  1. Actualice cada uno de los RPM del directorio que no fueron actualizados anteriormente, debido a que el RPM srvadmin-hapi no pudo actualizarse anteriormente, ejecutando el siguiente comando:

rpm -Uhv srvadmin-hapi*.rpm srvadmin-isvc*.rpm srvadmin-omhip*.rpm srvadmin-<cualesquier_otros_paquetes>*.rpm

Si ya está instalado en el sistema el RPM de OpenIPMI para una versión del archivo controlador OpenIPMI que cumpla con la versión mínima del archivo controlador OpenIPMI requerido por Server Administrator, pero los módulos de controlador de OpenIPMI aún no han sido generados e instalados para la versión dada del archivo controlador OpenIPMI debido a que el RPM kernel-source se necesita pero no está instalado actualmente, el RPM srvadmin-hapi no se podrá actualizar y mostrará un mensaje de error. El mensaje de error le indicará que debe instalar el RPM kernel-source y generar el archivo controlador de OpenIPMI. En este caso, todos los RPM que dependen del RPM srvadmin-hapi tampoco podrán actualizarse. Usted deberá instalar el RPM kernel-source, y generar e instalar los módulos de controlador de OpenIPMI, realizando los siguientes pasos adicionales:

  1. Para los sistemas que ejecutan Red Hat Enterprise Linux (versión 3), instale una copia del RPM kernel-source adecuada para el núcleo que está ejecutándose, a partir de los CD de instalación del sistema operativo (o de alguna otra fuente). En sistemas que ejecutan Red Hat Enterprise Linux (versión 4) y SUSE Linux Enterprise Server, los RPM kernel*-devel proporcionan la fuente necesaria para generar módulos de núcleo.

  2. Instale el RPM kernel-source o los RPM kernel*-devel ejecutando el siguiente comando desde donde encontró o copió los RPM:

rpm -ihv kernel*.rpm

  1. Navegue al directorio srvadmin/linux/supportscripts.

  2. Ejecute la secuencia de comandos de shell srvadmin-openipmi.sh como se muestra a continuación. Esto generará e instalará los módulos de controlador de OpenIPMI.

sh srvadmin-openipmi.sh install

Intente actualizar nuevamente el RPM srvadmin-hapi y los RPM que no se pudieron actualizar debido a que dependen del RPM srvadmin-hapi. Realice los pasos descritos en esta sección.

Uso de la secuencia de comandos de shell srvadmin-install

  3. Después de que el CD se monte, navegue al subdirectorio que contiene la secuencia de comandos de shell en el CD, escribiendo uno de los siguientes comandos:

NOTA: Justo antes de que se actualicen los RPM de Server Administrator, la secuencia de comandos de shell srvadmin-install.sh ejecuta automáticamente la secuencia de comandos de shell srvadmin-openipmi.sh para asegurarse haya una versión aceptable del archivo controlador OpenIPMI ya instalada en el sistema. Si no se encuentra una versión aceptable, la secuencia de comandos de shell srvadmin-install.sh ejecutará la secuencia de comandos de shell srvadmin-openipmi.sh para intentar la instalación de una versión aceptable. Si el intento falla por cualquier motivo, aparecerá un mensaje de error correspondiente con instrucciones que describen cómo resolver el error. Después de resolver los errores, debe ejecutar nuevamente la secuencia de comandos de shell srvadmin-install.sh para actualizar los componentes de Managed System.

Desinstalación de Managed System Software

Puede desinstalar el software Managed System desde la línea de comandos de Red Hat Enterprise Linux o SUSE Linux Enterprise Server. Además, se puede realizar una desinstalación desatendida en varios sistemas a la vez.

Requisitos previos para desinstalar Managed System Software

Se debe iniciar sesión como root.

Desinstalación del software Managed System desde la línea de comandos de Red Hat Enterprise Linux o SUSE Linux Enterprise Server.

Cuando se instala Server Administrator, también se instala una secuencia de comandos de desinstalación. Usted puede ejecutar la secuencia de comandos si escribe srvadmin-uninstall.sh y después presiona <Entrar>.

NOTA: Cuando se ejecuta, srvadmin-uninstall.sh no intentará desinstalar el archivo controlador OpenIPMI aun cuando el archivo controlador OpenIPMI haya sido instalado o actualizado durante alguna ejecución anterior de la secuencia de comandos srvadmin-install.sh o la secuencia de comandos srvadmin-openipmi.sh.

Desinstalación personalizada de componentes específicos

Es posible desinstalar componentes individuales de Dell OpenManage sin desinstalar todo Dell OpenManage. Los siguientes son ejemplos:

Para desinstalar únicamente el Web Server, use el comando:

rpm –e `rpm -qa | grep srvadmin-iws`

Para desinstalar diagnósticos, use el comando:

rpm –e `rpm -qa | grep srvadmin-old`

Para desinstalar el almacenamiento, use el comando:

rpm –e `rpm -qa | grep srvadmin-storage`

Utilización de Dell OpenManage con el software VMware ESX Server™

NOTA: La instalación de Dell OpenManage con software de VMware ESX Server requiere pasos especiales. Estos pasos varían según la versión de la versión de Dell OpenManage y de la versión de ESX Server; sólo se admite un número limitado de combinaciones.

Consulte la Guía de compatibilidad con sistemas VMware, localizada en el Centro de recursos, en www.dell.com/vmware, para determinar las versiones de software de ESX Server compatibles con esta versión de Dell OpenManage. Cada versión de ESX Server de Dell tiene una Guía de implementación de Dell VMware ESX Server asociada, que también está publicada en esta ubicación de web. Las instrucciones para instalar las versiones admitidas de Dell OpenManage disponibles en el momento de la publicación de esa versión de ESX Server se encuentran en la Guía de implementación de esa versión de ESX Server. Las instrucciones para instalar cualquier versión admitida publicada posteriormente de Dell OpenManage están publicadas en la misma ubicación, en documento separado y claramente etiquetado.

Instalación de Managed System Software usando software de implementación de terceros

Puede usar el software de iniciación de terceros, como Altiris Deployment Solution, para instalar Managed System Software en servidores Dell admitidos. Para distribuir e instalar el software Managed System usando Altiris, inicie la aplicación Altiris e importe el archivo OpenManage_Jobs.bin que se localiza en el CD Dell PowerEdge Installation and Server Management en \srvadmin\support\Altiris. Especifique la carpeta de trabajo a la cual lo desea importar. Es posible que tenga que modificar las tareas Ejecutar secuencia de comandos y Copiar archivo para que correspondan con el entorno de instalación utilizado. Una vez terminado, podrá programar la ejecución del trabajo en los sistemas Dell admitidos que son administrados desde Altiris Deployment Solution.
